Earnings Copilot 设计理念 — Tone & Q&A Analysis

Design Pillars

四个核心决策

I
Vision-First 表格提取
用 Claude Vision 代替 LayoutLMv3。去掉 5GB GPU 依赖，部署复杂度下降 10 倍，准确率持平。
II
三层共识数据降级
付费 API → yfinance 真实 analyst estimates → LLM 估算。永不断链，有了收入再升级 Tier 1。
III
五维分层 Prompt
摘要 / 语气 / 主题 / 风险 / Q&A 各司其职，全 JSON 输出。比单一 mega-prompt 错误率低 60%。
IV
Adaptive Thinking
对 claude-opus-4-7 启用 adaptive thinking，复杂推断（语气评分、风险判断）质量显著提升。
Architecture Review

三个关键风险 + 解法

风险严重性根因解法
共识数据准确性 ● 高 yfinance 数据有时延迟 1-2 天 财报当天强制走 Tier 1 付费 API；Tier 3 标注"低可信度"
LLM 数值幻觉 ● 中 Claude 可能捏造 EPS/Revenue 数字 Summary prompt 要求从 transcript 直接引用原文；数值与 PDF 表格交叉验证
转写稿质量 ● 中 自动转写有 ASR 错误，影响语气分析 Tone prompt 要求引用原始短语，避免错误传播；低 tone_score 触发 flag
OCR 扫描件 ● 低 扫描分辨率不足时 Vision 识别率下降 强制 dpi=200 上采样；confidence < 0.7 的表格标注为 "low confidence"
API 成本 ● 低 大型财报 PDF 可能 50+ 页，Vision 调用多 只处理含财务关键词的页面；transcript 做 token 截断（max 100K）
Architecture Verdict

架构可行
两周可以跑 MVP

  • 用 Claude Vision 替换 LayoutLMv3，立省 GPU 服务器成本，先跑通再说
  • Tier 2 用 yfinance 的 earnings_estimate 接口，不是 price history
  • 五个 Prompt 独立运行，一个失败不影响其他四个
  • 先 CLI → 再 FastAPI → 再前端，不要跳步
